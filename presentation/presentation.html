Static Islands in a Dynamic Sea

Carlton Gibson — Django Core, Steering Council

talks/01_static_islands_dynamic_sea

The Problem

Django models accumulate everything:

  • Display logic, queries, serialization, validation — all in one class
  • ORM fetches all fields by default
  • bookmark.user.username in a loop = one query per row
  • select_related / prefetch_related are opt-in and fragile

Why not just add types?

  • Django uses metaclasses, introspection, dynamic attributes
  • Type checkers can't follow objects.filter()
  • Forcing static types fights the framework

"Don't fight Django's dynamic nature. Build typed layers on top."

— Carlton Gibson

The Model


from django.contrib.auth.models import User
from django.db import models


class Bookmark(models.Model):
    url = models.URLField()
    title = models.CharField(max_length=255, blank=True)
    comment = models.TextField(blank=True)
    favourite = models.BooleanField(default=False)
    user = models.ForeignKey(User, on_delete=models.CASCADE)
    created_at = models.DateTimeField(auto_now_add=True)

Standard Django model. The FK on line 10 is where N+1 queries come from — accessing bookmark.user in a loop fires one query per bookmark.

The Solution: Data Shapes

Define what you need


import attrs
from mantle import Query

@attrs.define
class BookmarkData:
    url: str
    comment: str
    favourite: bool

# Mantle auto-generates only()
bookmarks = Query(
    Bookmark.objects.all(),
    BookmarkData
).all()  # type: list[BookmarkData]

What Mantle does for you

  • Auto-generates only() — fetches just declared fields
  • Auto-generates select_related() from nested shapes
  • Full type safety with attrs
  • DRF: shape_class replaces serializer_class

Nested Shapes: Solving N+1


@attrs.define
class UserData:
    username: str

@attrs.define
class BookmarkWithUser:
    url: str
    title: str
    favourite: bool
    user: UserData  # triggers auto-prefetch

Mantle sees the nested UserData and generates:


# What Mantle generates internally:
Bookmark.objects.only(
    "url", "title", "favourite",
    "user__username"

Advanced Indexing in Django & PostgreSQL

Haki Benita — 10+ years Django/PostgreSQL

talks/02_django_from_the_trenches

The Model: URL Shortener


from django.contrib.postgres.indexes import BrinIndex, HashIndex
from django.db import models
from django.db.models import F, Func, Q

class ShortUrl(models.Model):
    key = models.CharField(max_length=20)
    url = models.URLField(max_length=2048)
    hits = models.PositiveIntegerField(default=0)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        constraints = [
            # Covering index
            models.UniqueConstraint(
                fields=["key"],
                include=["url"],
            ),
        ]
        indexes = [
            # Function-based index
            models.Index(
                Func(F("url"), function="SUBSTRING",
                    template="%(function)s(%(expressions)s from '.*://([^/]*)')"),
                name="shorturl_domain_fix",
            ),

            # Partial index
            models.Index(
                fields=["id"],
                condition=Q(hits=0),
                name="shorturl_unused_pix",
            ),

            # Hash index
            HashIndex(
                fields=["url"],
                name="shorturl_url_hix",
            ),

            # BRIN index
            BrinIndex(
                fields=("created_at",),
                pages_per_range=4,
                name="shorturl_created_bix",
            ),
        ]
One model, five index types. Let's zoom into each one.

Covering Index

Use case: Redirect /r/abc123/ to the full URL

Problem: Normal B-Tree finds the key, then goes to the table to fetch the URL. Extra disk read (heap fetch).


models.UniqueConstraint(
    fields=["key"],
    name="shorturl_key_uk",
    include=["url"],  # stored in index leaf
)

Line 4: include=["url"] stores the URL inside the index itself.

0 heap fetches

Index-Only Scan

How it works

Lookup finds key in the index → URL is right there in the leaf block → zero table access.

Trade-off: doubles storage for the included field.

Function-Based Index

Use case: Find all short URLs pointing to github.com

Problem: Without index, PostgreSQL runs a regex on every row, every query.


# The query (alias, not annotate!)
ShortUrl.objects.alias(
    domain=Func(F("url"), function="SUBSTRING",
        template="...from '.*://([^/]*)'..."),
).filter(domain="github.com")

# The index (same expression!)
models.Index(
    Func(F("url"), function="SUBSTRING",
        template="...from '.*://([^/]*)'..."),
    name="shorturl_domain_fix",
)

3s → 51ms

60x faster

Key insight

The regex runs once at insert, result stored in index. Expression in query must be identical to index.

Bonus: use alias() not annotate() — avoids duplicate computation in SQL.

Partial Index

Use case: Find unused short URLs (hits=0) for cleanup

Problem: A full B-Tree index on hits covers all 10,000 rows (7 MB). But you only query hits=0.


models.Index(
    fields=["id"],
    condition=Q(hits=0),  # only these rows
    name="shorturl_unused_pix",
)

Line 3: Only rows where hits=0 are indexed.

7 MB → 88 KB

99% smaller

Biggest quick win

Nullable FKs: Django auto-creates full indexes on every FK. Set db_index=False, add a partial index with condition=Q(field__isnull=False).

One team freed 20 GB of index space this way.

Hash Index

Use case: Reverse lookup — "is this URL already shortened?"

Problem: B-Tree stores the full URL text in the index. Long URLs = huge index.


from django.contrib.postgres.indexes import HashIndex

HashIndex(
    fields=["url"],
    name="shorturl_url_hix",
)

47 MB → 32 MB

Constant-size hash

Limitations

  • Equality only — no range queries, no ordering
  • No multi-column, no FK references
  • Best for: long strings, URLs, UUIDs

BRIN Index

Use case: Find short URLs created in the last 30 days

Problem: B-Tree on created_at is 2208 KB with one entry per row.


from django.contrib.postgres.indexes import BrinIndex

BrinIndex(
    fields=("created_at",),
    pages_per_range=4,
    name="shorturl_created_bix",
)

Stores only min/max per block of pages. Works because rows are inserted in time order.

SizeSpeed
B-Tree2208 KB2.2 ms
BRIN64 KB5.2 ms

35x smaller, slightly slower

Trade speed for space. Works when data is naturally ordered on disk. Check with pg_stats.correlation.

Reliable Django Signals

Haki Benita — hakibenita.com

talks/03_reliable_django_signals

Why Django Signals Are Unreliable

3 failure modes:

  • Receiver crashes sender — signal.send() propagates exceptions; one bad receiver rolls back the entire transaction
  • Side effects can't undo — receiver sends SMS, then txn rolls back. User gets a message for nothing
  • The gap — move signal outside atomic()? Process crashes after commit but before signal → receivers never execute

The scenario:

Payment succeeds → signal fires → order updates

Receiver fails: payment rolls back too

Signal outside txn: crash = lost signal

send_robust() swallows errors but still runs inside txn

There's no position for signal.send() that's both safe and reliable.

The Code: ReliableSignal


class ReliableSignal(DjangoSignal):
    def send_reliable(self, sender, **named):
        if not self.receivers:
            return
        if self.sender_receivers_cache.get(sender) is NO_RECEIVERS:
            return
        sync_receivers, _ = self._live_receivers(sender)
        for receiver in sync_receivers:
            execute_task_signal_receiver.enqueue(
                receiver_qualname=callable_to_qualname(receiver),
                named=named,
            )

Step 1

Extend Django's Signal class

Step 2

Guard: skip if no receivers

Step 3

Enqueue each receiver as a database task

The Outbox Pattern

Payment Model


@classmethod
def set_status(cls, id, *, succeeded):
    with transaction.atomic():
        payment = cls.objects.select_for_update().get(id=id)
        payment.status = "succeeded" if succeeded else "failed"
        payment.save()

        signals.payment_process_completed.send_reliable(
            sender=None,
            payment_process_id=payment.id,
        )
    return payment

What happens

  • Line 3: Everything inside one transaction
  • Lines 4-6: Lock row, update, save
  • Lines 8-11: send_reliable writes task to DB inside the same txn

The magic

If txn rolls back → task row disappears.
If txn commits → worker picks up the task.
Failed receivers can be retried.

The Receiver


@staticmethod
@receiver(payment.signals.payment_process_completed)
def on_payment_completed(payment_process_id, **kwargs):

    with transaction.atomic():
        order = Order.objects.select_for_update().get(
            payment_process_id=payment_process_id
        )
        match order.payment_process.status:
            case "succeeded": order.status = "completed"
            case "failed": order.status = "cancelled"
        order.save()

The receiver runs in its own transaction, in a separate worker process. Completely isolated from the sender.

The Result

Decoupled

At-least-once delivery

Retryable

Django 6 Tasks Framework

Uses django-tasks with a database backend. Tasks are committed atomically with data changes. Worker: python manage.py db_worker

This is exactly the transactional outbox pattern. Works with Celery too — Django 6's tasks framework just makes it convenient.

Single-Field Inline Editing

The idea: Edit one field at a time, inline, no page reload.


class SingleFieldFormMixin:
    """Marker mixin. All subclasses
    are auto-discovered via
    __subclasses__()"""

class TitleForm(SingleFieldFormMixin, ModelForm):
    class Meta:
        model = Book; fields = ["title"]

One ModelForm per field. Registry is implicit.

How it works

  • Click a value → HTMx GET swaps it for a form
  • Submit → HTMx POST saves, swaps back to value
  • Server stays 100% Django forms
  • JSON-backed fields = editable fields without migrations

HTMx Does the Swap


<!-- Display mode -->
<span hx-get="/books/1/edit/title/"
      hx-target="closest div"
      hx-swap="innerHTML">
  The Great Gatsby
</span>

<!-- Edit mode (returned by GET) -->
<form hx-post="/books/1/edit/title/"
      hx-target="closest div">
  <input name="title" value="The Great Gatsby">
</form>

No new library needed. HTMx handles the swap, Django handles the form. Converted Argus (production alert system) from React SPA to this pattern.

Role-Based Access Control in Django

Gergő Simonyi — authentik (open-source IdP)

talks/08_role_based_access_control_in_django

Beyond Django's Built-in Auth

Django gives you:

  • Model-level perms: user.has_perm("books.change_book")
  • Guardian adds object-level perms

Customers wanted:

  • Group hierarchy — parent perms flow to children
  • JIT access — grant for N hours, auto-expire
  • Object-level scoping — editor on this book
  • Custom perms — copy_book, archive_invoice

The three moves

1. Strip Django's authz

Subclass ModelBackend → return empty/False for all authz. Keep authentication.

2. Single RBAC backend

One has_perm(user, perm, obj) method answers everything.

3. Cache the hierarchy

Transitive closure table — one row per (ancestor, descendant, depth).

The RBAC Models


class Role(models.Model):
    name = models.CharField(max_length=100, unique=True)
    parent = models.ForeignKey("self", null=True, on_delete=models.SET_NULL)
    permissions = models.ManyToManyField(Permission, blank=True)

class RoleAncestry(models.Model):  # Closure cache
    ancestor   = models.ForeignKey(Role, related_name="descendant_links")
    descendant = models.ForeignKey(Role, related_name="ancestor_links")
    depth      = models.PositiveIntegerField()

class UserRole(models.Model):
    user         = models.ForeignKey(AUTH_USER_MODEL, ...)
    role         = models.ForeignKey(Role, ...)
    content_type = models.ForeignKey(ContentType, null=True)  # scope
    object_id    = models.PositiveIntegerField(null=True)     # scope
    expires_at   = models.DateTimeField(null=True)            # JIT

Role

Parent FK (hierarchy) + permissions M2M

RoleAncestry

Closure cache (ancestor, descendant, depth)

UserRole

Assignment + scope (GenericFK) + JIT (expires_at)

The Closure Table

Role hierarchy


viewer       → view_book
  └ editor  → change_book (inherits view)
      └ librarian → add, delete (inherits all)

Closure rows generated


(viewer, viewer, 0)       self-link
(editor, editor, 0)       self-link
(viewer, editor, 1)       viewer is ancestor of editor
(viewer, librarian, 2)    2 hops up
(editor, librarian, 1)    1 hop up
(librarian, librarian, 0) self-link

Rebuilt on save/delete via signals.

Self-links (R, R, 0) mean one join covers "role or ancestors".

In authentik (production)

PostgreSQL materialized view + pgtrigger for concurrent refresh. Same concept, better performance at scale.

Single-Query Permission Check


def has_perm(self, user_obj, perm, obj=None):
    if not user_obj.is_active: return False
    if user_obj.is_superuser: return True
    app_label, codename = perm.split(".", 1)

    direct_roles = self._direct_role_ids(user_obj, obj)  # honors JIT + scope

    # Any ancestor of a directly-held role holds the permission?
    return Role.objects.filter(
        descendant_links__descendant__in=direct_roles,
        permissions__content_type__app_label=app_label,
        permissions__codename=codename,
    ).exists()

~10 lines. One query. No recursion. The closure table removed the depth from the problem.

What's in Your Dependencies?

Mateusz Belczowski — Profil Software, Poland

talks/10_whats_in_your_dependencies

Real Supply Chain Attacks on PyPI

Typosquatting

jeIlyfish (capital I, not L) — stole SSH/GPG keys. On PyPI for 1 year.

Phishing Maintainers

Fake "PyPI validation" emails (Aug 2022). Credentials stolen, malicious versions published.

CI/CD Hijacking

Codecov (2021): 23K+ customers. YOLO (2024): crypto miner via GitHub Actions.

Account Hijacking

ctx: attacker bought expired domain → password reset → AWS key theft. ~27K downloads.

Dependency confusion — Alex Birsan breached 35+ companies (Apple, Microsoft, Tesla). $130K+ in bounties.

The Code: Pin with Hashes

Version pin (not enough)


django==5.2
requests==2.32.3

Tells pip which version. But if the file on PyPI is replaced, pip installs it.

Hash pin (file integrity)


django==5.2 \
    --hash=sha256:abc123...
pip install --require-hashes -r requirements.txt

Tells pip which exact file. Hash mismatch = installation fails.


# Generate hashes with uv (recommended)
uv pip compile requirements.in --generate-hashes -o requirements.txt

# Scan for known vulnerabilities
pip-audit -r requirements.txt

The Defense Toolkit

Scan

  • pip-audit (OSV database)
  • Run in CI on every PR

Pin with Hashes

  • --require-hashes
  • uv pip compile --generate-hashes

Evaluate First

  • deps.dev, Scorecard, Socket.dev
  • Bus factor, last release

Trusted Publishing

OIDC tokens replace API tokens. No token to steal.

Isolate Install

pip install = full access. Virtualenvs = zero sandbox. Use Docker.

Monitor

Socket.dev, Dependabot, Renovate. Continuous.

PEP 740 Attestations

  • Trusted Publishing replaces long-lived API tokens with short-lived OIDC tokens tied to CI providers
  • PEP 740 (GA Nov 2024): Sigstore-based proof of where a package was built
  • 132,000+ packages have attestations (as of March 2026)
  • uv defaults to "first-index" resolution — prevents dependency confusion attacks

AI tools make it worse

LLMs sometimes suggest installing packages that don't exist. Attackers can register those names. Verify before pip install.

Action Items

Do today:

  • Add pip-audit to CI pipeline
  • Switch to hash-pinned requirements
  • Enable Trusted Publishing on PyPI
  • Review pip install in Dockerfiles

Before adding a dependency:

  • How many maintainers? (bus factor)
  • When was the last release?
  • Does it use Trusted Publishing?
  • Can you vendor it instead?

Fewer dependencies = fewer attack vectors.

Is It Time for a Django Admin Rewrite?

Emma Delescolle — Django Steering Council, LevIT Belgium

talks/11_is_it_time_for_django_admin_rewrite

Why the Admin Needs Rethinking

  • Framework inside a framework — own patterns, view system, template hierarchy
  • Package collisions — django-import-export, django-filter, themes all override templates
  • No plugin architecture — every package reinvents hooks
  • 20 years of legacy — predates CBVs, dataclasses, type hints
  • Customization wall — row-level actions, form layouts need deep overrides
  • #1 feature request area on Django's tracker

"We should rethink the admin."

— Django Issue #70

Stock Admin vs Admin-Deux

Stock Admin


@admin.register(Product)
class ProductAdmin(admin.ModelAdmin):
    list_display = [
        "name", "sku",
        "category", "price",
    ]

Strings only. No filtering, ordering, or styling without overrides.

Admin-Deux


from djadmin import register, Column
from djadmin.dataclasses import Filter

@register(Product)
class ProductAdmin(ModelAdmin):
    list_display = [
        "name",
        Column("sku", label="SKU Code",
               classes="font-mono"),
        Column("category",
               filter=True, order=True),
        Column("price", order=True,
               filter=Filter(
                   lookup_expr=["gte","lte"])),
    ]
