# Django Transaction History (covered in talk)

- Django 0.95 (July 2006) — first transaction support, commit after every save/delete
  - Release notes: https://docs.djangoproject.com/en/5.2/releases/0.95/
- Django 1.6 (November 2013) — autocommit by default, atomic() introduced, commit_on_success deprecated
  - Release notes: https://docs.djangoproject.com/en/5.2/releases/1.6/
- Django 3.2 (April 2021) — durable parameter added to atomic()

# Key Concepts Explained

- Transactions: BEGIN / COMMIT / ROLLBACK — atomicity at SQL level
- Savepoints: partial rollback within a transaction (used by nested atomic blocks)
- Durability: guarantee that committed data won't be rolled back by outer context
- on_commit: register callbacks to fire after transaction commits
- autocommit mode vs default mode (implicit begin vs implicit commit)

# Q&A Highlights

- Audience asked about getting subatomic into Django core — speakers said they've started writing a proposal but have unsolved problems (namespace clashes, deferred constraints in tests)
- Works alongside existing atomic() blocks during gradual migration
- transaction_required used orders of magnitude more than savepoint at Kraken
- Nested savepoints in PostgreSQL cause performance issues (switches from memory to disk at depth)
- Kraken uses internal wrappers with metrics to measure transaction depth
- Only tested against PostgreSQL so far, but no DB-specific SQL used — plan to support all backends
