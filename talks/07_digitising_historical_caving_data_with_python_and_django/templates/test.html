Talk 07 -- Digitising Historical Caving Data

Sixty years of caving incidents trapped in scanned PDFs. This pipeline turns them into a structured, queryable Django database using OCR, LLMs for cleanup, a self-check pattern, a custom date field, and a tree for locations.

The Five-Stage Pipeline

Every incident report goes through these steps. Each is an independent, pluggable operation class.

1. OCR (Docling) → 2. Segmentation → 3. LLM Rewrite → 4. LLM Critic → 5. Structuring

This demo implements stages 3-5 with fake (deterministic) LLM stubs. The architecture -- pluggable steps, dependency tracking, failure recording -- is what matters, not the LLM prompts.

1

Pluggable Processing Steps

What it proves: Every pipeline step is an Operation subclass with a tiny contract: should_run(incident) + run(incident). A @register decorator adds it to the pipeline. Dependencies are declared via requires = [OtherOp]. New step = new file. No central list, no if-ladder. 
# incidents/operations/base.py
class Operation:
    requires: list[type["Operation"]] = []

    def should_run(self, incident) -> bool:
        return True

    def run(self, incident) -> None:
        raise NotImplementedError

def register(op_cls):            # decorator: adds to pipeline
    _REGISTRY.append(op_cls)
    return op_cls
# incidents/operations/classify_severity.py
@register
class ClassifySeverity(Operation):
    requires = [FakeLLMRewrite]    # won't run until rewrite succeeds

    def run(self, incident):
        text = incident.cleaned_text.lower()
        for keyword, severity in KEYWORDS:
            if keyword in text:
                incident.severity = severity
                incident.save(update_fields=["severity"])

The runner iterates operations x incidents, records an OperationRun row per attempt, and respects dependencies.

Self-Check: The Critic Pattern

What it proves: A second LLM pass grades the first. If the output still looks bad, the step records FAILED without crashing the pipeline. Incident #8 has unresolvable ??? placeholders -- the critic catches this and flags it. The pipeline keeps running; failures surface in the dashboard. 
# The rewrite step cleans OCR artifacts but CANNOT resolve ???
raw:     "Party of three descended ??? pitch; ??? injury sustained."
cleaned: "Party of three descended ??? pitch; ??? injury sustained."

# The critic checks for suspicious patterns:
SUSPICIOUS = [
    re.compile(r"\?{2,}"),           # unreadable-glyph placeholder
    re.compile(r"\s{2,}"),           # leftover double whitespace
]

# If any match, the critic raises -> recorded as FAILED
raise ValueError("cleaned_text still contains: \\?{2,}")

Custom FuzzyDate Model Field

What it proves: Historical dates come in many precisions: "1971", "August 1985", "Autumn 1996", "15 March 2024". A plain DateField forces false specificity. FuzzyDate stores precision alongside the date in a single, lexicographically-sortable string. The three Django hooks -- to_python / from_db_value / get_prep_value -- are the whole custom-field pattern in ~30 lines. 
# The three hooks that make a custom model field:

class FuzzyDateField(models.CharField):
    def from_db_value(self, value, expression, connection):
        # DB -> Python (on every query row)
        return FuzzyDate.from_storage(value)

    def to_python(self, value):
        # Forms/fixtures/assignments -> Python
        return FuzzyDate.from_storage(value)

    def get_prep_value(self, value):
        # Python -> DB (on every save)
        return value.to_storage()

Storage Format

Each date stores as a single TEXT column. The format is lexicographically sortable, so chronological ordering works without any special SQL.

Tree-Structured Locations (django-treebeard)

What it proves: Incident locations have variable depth. Some reports say "Arizona" (state), others say "Carlsbad Caverns, New Mexico, USA" (region). A flat schema with nullable country/state/region/cave columns cannot represent this cleanly. MP_Node (Materialized Path) makes each node store its path as a compact string. get_descendants() gives you "all incidents in the USA" in a single query. 
from treebeard.mp_tree import MP_Node

class Location(MP_Node):
    name = models.CharField(max_length=100)
    level = models.CharField(max_length=20, choices=LocationLevel.choices)
    node_order_by = ["name"]

# Building the tree:
usa = Location.add_root(name="USA", level="country")
nm = usa.add_child(name="New Mexico", level="state")
nm.add_child(name="Carlsbad Caverns", level="region")

# All incidents anywhere in the USA (one query!):
Incident.objects.filter(location__in=usa.get_descendants())
