Talk 03 — Reliable Django Signals

When something important happens (payment succeeds), you need to tell other parts of your app. Django signals can lose that message. This page shows you how and why.

Setup

The demo uses two models: Order (starts as pending_payment) and PaymentProcess (starts as initiated). When a payment completes, the order should update. The question is how to deliver that message reliably.

1

signal.send() inside the transaction

What happens: You call signal.send() inside transaction.atomic(). Django calls every receiver right there, inline. If any receiver raises an exception (email service down, database error, anything), the exception propagates up and the entire transaction rolls back — including the payment that was perfectly valid.

The problem: A broken receiver kills the sender. Your payment was fine, but it gets rolled back because some unrelated code failed.

payment.save() + signal.send() → receiver crashes! → ROLLBACK (everything) 
with transaction.atomic():
    payment.status = "succeeded"
    payment.save()

    # signal.send() calls receivers HERE, inside the transaction.
    # If a receiver raises, the entire transaction rolls back
    # — including the payment.save() above!
    signal.send(...)
2

signal.send() after the transaction — the gap

OK, so move the signal outside the transaction? Now the payment is committed first, then you call signal.send(). But what if the server crashes between commit and send? The payment is saved in the database (it's committed), but the signal never fires. The order stays pending_payment forever. Nobody knows.

The problem: There's a gap between "data saved" and "signal sent." Anything that happens in that gap (crash, restart, timeout) means the message is lost.

payment.save() → COMMIT → CRASH → signal.send() never happens 
with transaction.atomic():
    payment.status = "succeeded"
    payment.save()
# Transaction committed. Payment is saved.

# Server crashes RIGHT HERE.
# signal.send() never runs. Order stays pending forever.
signal.send(...)

Polling fallback: A cron job periodically checks: "are there orders stuck in pending_payment whose payment already succeeded?" If so, fix them. This catches anything the signal missed. Click "Poll Orders" above to try it.

3

send_reliable() — the transactional outbox

The fix: Instead of calling receivers, write a task row in the database inside the same transaction as the payment. The task row and the payment status change are committed together — or neither.

If the transaction commits → both the payment AND the task are saved. A background worker will find the task and run the receiver. Even if the server crashes right after commit, the task is already in the DB.

If the transaction rolls back → both the payment AND the task disappear. No phantom emails, no orphaned tasks.

Same transaction = they live and die together.

payment.save() + task row inserted → COMMIT → worker picks up task → receiver executes 
with transaction.atomic():
    payment.status = "succeeded"
    payment.save()

    # Don't call receivers — save a task in the SAME transaction.
    # Task row is committed atomically with the payment.
    signal.send_reliable(...)

Try it: Click "Complete Payment" — the order shows QUEUED (not STUCK). The payment is done, and a task is safely in the database. Then click "Process Tasks" to simulate the worker — the order updates. In production, db_worker runs continuously and processes tasks automatically.

Key Code: ReliableSignal

Replaces signal.send(). Instead of calling receivers inline, enqueues a task per receiver inside the same DB transaction.

class ReliableSignal(DjangoSignal):

    def send_reliable(self, sender, **named):
        if not self.receivers:
            return

        sync_receivers, _ = self._live_receivers(sender)

        for receiver in sync_receivers:
            # Each receiver becomes a task row in the database.
            # Written inside the caller's transaction.atomic().
            execute_task_signal_receiver.enqueue(
                receiver_qualname=callable_to_qualname(receiver),
                named=named,
            )
