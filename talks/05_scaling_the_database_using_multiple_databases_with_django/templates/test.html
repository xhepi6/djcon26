Challenge #0

Replication Lag

Before we even talk about routers: replicas are always behind the primary. When you write a row to the primary, the replica doesn't see it until replication catches up. In production (AWS RDS, etc.) lag is usually single-digit milliseconds. Here we simulate it with two SQLite files and a manual sync_replica command.

INSERT author → Primary: 4 rows → Replica: 3 rows (stale!) → sync_replica → Replica: 4 rows 
# Two database aliases in settings.py
DATABASES = {
    "default": {           # primary -- handles writes
        "ENGINE": "...",
        "NAME": "primary.sqlite3",
    },
    "replica": {           # replica -- handles reads (when in sync)
        "ENGINE": "...",
        "NAME": "replica.sqlite3",
        "TEST": {"MIRROR": "default"},  # tests share one DB
    },
}

Challenge #1

Reads Inside Transactions

Inside transaction.atomic(), writes go to the primary (via db_for_write). But reads go to the replica (via db_for_read). The replica can't see uncommitted data from the primary's in-progress transaction. So a count after a create returns 0 with a naive router. The fix: check transaction.get_autocommit("default") in the router. When it returns False, we're inside a transaction — route reads to the primary too.

atomic() → CREATE author → COUNT(*) → Naive: replica says 0
atomic() → CREATE author → COUNT(*) → Smart: primary says 1 
# The fix in db_for_read:
def db_for_read(self, model, **hints):
    # Inside a transaction? Read from primary (see own writes)
    if not transaction.get_autocommit(using="default"):
        return "default"
    return "replica"

Challenge #2

get_or_create Always Hits Primary

Django's get_or_create always routes through the write connection, even when the row already exists. It does this to guarantee read-your-writes consistency. That means every call hits the primary, even on the happy path where the row is already there. The fix: a replica_aware_get_or_create that tries .get() first (routed to replica), then falls back to the real get_or_create only on miss.

get_or_create("Jake") → Always PRIMARY (even if Jake exists)
replica_aware: .get("Jake") → REPLICA (happy path) | DoesNotExist? get_or_create → PRIMARY (fallback) 
# helpers.py
def replica_aware_get_or_create(manager, defaults=None, **kwargs):
    try:
        return manager.get(**kwargs), False     # replica (via router)
    except manager.model.DoesNotExist:
        return manager.get_or_create(defaults=defaults or {}, **kwargs)  # primary

Challenge #3

GenericRelation .update() Routes to Replica

Django ticket #36389: calling instance.generic_relation.update(...) is a write, but the ORM routes it through the read connection. On a real replica this would fail (read-only). The fix: at startup, walk the model registry to find models with GenericForeignKey or GenericRelation, and force them to default in the router.

revision.mains.update(text=...) → Naive: UPDATE on REPLICA
revision.mains.update(text=...) → Smart: UPDATE on PRIMARY 
# The fix in db_for_read -- detect GFK models at startup:
def _models_with_generic_fields():
    out = set()
    for model in apps.get_models():
        for field in model._meta.get_fields():
            if isinstance(field, (GenericForeignKey, GenericRelation)):
                out.add(model)
                break
    return out

class PrimaryReplicaRouter:
    @cached_property
    def _gfk_models(self):
        return _models_with_generic_fields()

    def db_for_read(self, model, **hints):
        if model in self._gfk_models:
            return "default"   # force GFK models to primary
        ...
