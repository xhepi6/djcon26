What you're looking at

Click any row in the permission matrix to see the actual query Django ran. Scroll to section 3 to watch just-in-time expiry flip a check from True to False live.

The problem & the solution

The problem

  • Django ships model-level permissions: user.has_perm("app.change_book") — granted via user or group.
  • django-guardian adds object-level permissions on top (generic FK to the target row).
  • Neither handles: group hierarchy (permissions flow down a parent→child chain), just-in-time access (grant admin for an hour), delegation (share my access with Bob), or custom non-CRUD permissions (copy_book, approve_payout).
  • Bolting a Role layer on top of Guardian means every check walks user → group → parent → ... → role → permission — N queries per check. Fine in demos, dead in prod.

The solution

  • Strip Django's authz half — ModelBackendNoAuthz keeps authenticate() but zeroes every has_perm / get_*_permissions so the chain always falls through.
  • One custom backend (RBACBackend) answers has_perm directly against RBAC tables.
  • Transitive-closure cache: one row per (ancestor, descendant, depth), including self-links at depth 0. A permission check joins through it — one query, any depth.
  • Object scope & JIT are just columns on UserRole: content_type/object_id for the object, expires_at for JIT. Custom perms are plain Meta.permissions.

Role hierarchy → transitive closure

The hierarchy (left) is a DAG of Role.parent edges. The closure (right) is the cache: every ancestor-descendant pair the hierarchy implies, plus the depth between them. A permission check joins through this table, so depth disappears from the problem.

Depth-0 rows (italicized) are self-links — "role R is its own ancestor at distance 0". They let the same join cover both "this role" and "any ancestor of this role". Rebuilt on every Role.save() / Role.delete() via signal. In production with Postgres: a materialized view + pgtrigger.

Users, assignments & the permission matrix

Each UserRole row binds one user to one role — optionally scoped to a single object (content_type + object_id) and/or time-bounded (expires_at).
