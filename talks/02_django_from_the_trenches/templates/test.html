Step 0: Your Debugging Toolbox

Before optimizing, you need to see what PostgreSQL is doing. Three commands:

print(qs.query) See the raw SQL that Django generates
print(qs.explain()) See the execution plan (what PostgreSQL plans to do, without running it)
print(qs.explain(analyze=True)) Run it and get real timing and actual row counts

The workflow: measure baseline, add index, run explain(analyze=True), compare. Every card below shows real results from this workflow.

Index Sizes at a Glance

Same table, same 10,000 rows. The right index type can be 100x smaller. More indexes = slower writes + more disk. Pick wisely.

1. Covering Index

...

Use case: Someone visits /r/abc123/ and we need to redirect them to the full URL. We look up the key and only need the URL back. Normally PostgreSQL finds the key in the index, then goes to the table to fetch the URL. With include=["url"], the URL is stored inside the index itself -- it never touches the table. Zero heap fetches. 

from django.db import models

class ShortUrl(models.Model):
    key        = models.CharField(max_length=20)
    url        = models.URLField(max_length=2048)
    hits       = models.PositiveIntegerField(default=0)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        constraints = [
            models.UniqueConstraint(
                fields=["key"],
                include=["url"],  # stored in index leaf, not just indexed
                name="shorturl_key_uk",
            ),
        ]

2. Function-Based Index

                ...
            

            

                Use case: Find all short URLs pointing to github.com.
                The domain is buried inside the url field -- we need a regex to extract it.
                Without this index, PostgreSQL runs that regex on every row, every query.
                With it, the regex result is computed once on insert and stored in the index.
            


from django.db import models
from django.db.models import F, Func

class ShortUrl(models.Model):
    key        = models.CharField(max_length=20)
    url        = models.URLField(max_length=2048)
    hits       = models.PositiveIntegerField(default=0)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        indexes = [
            models.Index(
                Func(
                    F("url"),
                    function="SUBSTRING",
                    template="%(function)s(%(expressions)s from '.*://([^/]*)')",
                ),
                name="shorturl_domain_fix",
            ),
        ]

3. Partial Index

                ...
            

            

                Use case: Find short URLs that have never been clicked (hits=0)
                so we can clean them up or send reminders. About 30% of our 10,000 URLs are unused.
                A full index would cover all 10,000 rows. This one only indexes the ~3,000 that match hits=0.
                Same speed, fraction of the disk space.
                Best candidates: nullable FKs, boolean flags, status fields.
            


from django.db import models
from django.db.models import Q

class ShortUrl(models.Model):
    key        = models.CharField(max_length=20)
    url        = models.URLField(max_length=2048)
    hits       = models.PositiveIntegerField(default=0)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        indexes = [
            models.Index(
                fields=["id"],
                condition=Q(hits=0),  # only rows where hits=0
                name="shorturl_unused_pix",
            ),
        ]

4. Hash Index

                ...
            

            

                Use case: Reverse lookup -- someone pastes a long URL like
                https://docs.djangoproject.com/en/5.1/topics/... and we check if it's already shortened.
                B-Tree stores the full URL text in the index -- long URLs = big index.
                Hash stores a fixed-size hash regardless of URL length.
                Only works for exact = lookups. No LIKE, no range queries, no ORDER BY.
            


from django.db import models
from django.contrib.postgres.indexes import HashIndex

class ShortUrl(models.Model):
    key        = models.CharField(max_length=20)
    url        = models.URLField(max_length=2048)
    hits       = models.PositiveIntegerField(default=0)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        indexes = [
            HashIndex(
                fields=["url"],
                name="shorturl_url_hix",
            ),
        ]

5. BRIN Index

                ...
            

            

                Use case: Find all short URLs created in the last 30 days for a monthly report.
                B-Tree creates one entry per row (big). BRIN stores just the min/max per block of pages (tiny -- 24 KB).
                It works here because rows are inserted in time order, so physical disk order matches created_at order.
                If your data were randomly ordered, BRIN would be useless.
            


from django.db import models
from django.contrib.postgres.indexes import BrinIndex

class ShortUrl(models.Model):
    key        = models.CharField(max_length=20)
    url        = models.URLField(max_length=2048)
    hits       = models.PositiveIntegerField(default=0)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        indexes = [
            BrinIndex(
                fields=("created_at",),
                pages_per_range=4,  # lower = more precise, bigger index
                name="shorturl_created_bix",
            ),
        ]

6. alias() vs annotate()

            

                Use case: Same domain search as #2, but look at the SQL difference.
                annotate() computes SUBSTRING and puts it in both SELECT and WHERE -- it does the work twice.
                alias() only puts it in WHERE. Use alias when you're filtering but don't need the value in the results.
