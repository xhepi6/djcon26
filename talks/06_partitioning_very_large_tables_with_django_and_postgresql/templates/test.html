The Problem

Why Partition? The 9-Billion-Row Table

A single table grows to 9 billion rows, 3 TB. Vacuuming takes 20 hours, runs daily, and blocks schema changes. You can't delete old data (app changes needed) or start fresh (defers the problem). Partitioning splits one logical table into many physical tables. Each partition is vacuumed, indexed, and queried independently. One 20-hour vacuum becomes thirty 40-minute vacuums running in parallel.

-- The parent is virtual: PARTITION BY RANGE on the partition key
CREATE TABLE workflows_workflowstep (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY,
    workflow_id BIGINT NOT NULL,
    name        VARCHAR(100) NOT NULL,
    status      VARCHAR(20) NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL,
    PRIMARY KEY (id, created_at)  -- partition key MUST be in every unique constraint
) PARTITION BY RANGE (created_at);

-- Each child is an ordinary table with its own indexes + vacuum
CREATE TABLE workflows_workflowstep_2026_04
    PARTITION OF workflows_workflowstep
    FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');

Key Concept

Partition Pruning: The Performance Win

When a query's WHERE clause includes the partition key, PostgreSQL checks each partition's bounds and prunes the ones that can't match. A query for March 2026 data only scans the 2026_03 partition — the other 7+ partitions are never touched. Without the partition key in the filter, PostgreSQL must scan every partition (an Append over all children). 

# Django ORM: partition pruning is transparent to application code
# The partition key (created_at) is in the WHERE -- pruning happens
WorkflowStep.objects.filter(
    created_at__gte=date(2026, 3, 1),
    created_at__lt=date(2026, 4, 1),
    workflow_id=7,
)
# EXPLAIN shows: only workflows_workflowstep_2026_03 is scanned

# Without the partition key -- no pruning, scans everything
WorkflowStep.objects.filter(workflow_id=7)
# EXPLAIN shows: Append across ALL partitions

Leaky Abstraction

Django 5.2 Composite Primary Keys

PostgreSQL requires the partition key to be part of every unique constraint, including the primary key. A plain PRIMARY KEY (id) is rejected. You need PRIMARY KEY (id, created_at). Django 5.2's CompositePrimaryKey makes reads work transparently. But writes are leaky: CompositePrimaryKey doesn't combine with BigAutoField, so the id comes from a database-side IDENTITY sequence. ORM .create() can't skip the id column, so inserts use raw SQL. 

# models.py -- the model uses managed=False (SQL migration owns the schema)
class WorkflowStep(models.Model):
    pk = models.CompositePrimaryKey("id", "created_at")
    id = models.BigIntegerField()        # DB-side IDENTITY sequence
    created_at = models.DateTimeField()  # partition key
    workflow_id = models.BigIntegerField()
    name = models.CharField(max_length=100)
    status = models.CharField(max_length=20)

    class Meta:
        managed = False  # Django can't emit PARTITION BY RANGE

# Reads: ORM just works -- composite PK is transparent to SELECT
WorkflowStep.objects.filter(status="done").count()   # works
WorkflowStep.objects.order_by("created_at")[:10]     # works

# Writes: raw SQL because IDENTITY + CompositePK don't combine
with connection.cursor() as cur:
    cur.execute("""
        INSERT INTO workflows_workflowstep
            (workflow_id, name, status, created_at)
        VALUES (%s, %s, %s, %s)
    """, [7, "validate_meter", "pending", now])

Practical

Migration Strategy and Maintenance

Django's migration framework can't emit PARTITION BY RANGE. Use RunSQL in a migration to create the parent and initial partitions. The model is managed = False so Django doesn't fight you. For new partitions, use another RunSQL migration, or automate with cron / pg_partman. 

# migrations/0001_initial.py -- RunSQL creates everything
class Migration(migrations.Migration):
    initial = True
    operations = [
        migrations.RunSQL(
            sql="""
            CREATE TABLE workflows_workflowstep (
                ...
            ) PARTITION BY RANGE (created_at);

            CREATE TABLE workflows_workflowstep_2026_04
                PARTITION OF workflows_workflowstep
                FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');

            CREATE TABLE workflows_workflowstep_default
                PARTITION OF workflows_workflowstep DEFAULT;
            """,
            reverse_sql="DROP TABLE IF EXISTS workflows_workflowstep CASCADE;",
        ),
    ]

# Adding a new partition at runtime:
# python manage.py add_partition 2026-06
CREATE TABLE workflows_workflowstep_2026_06
    PARTITION OF workflows_workflowstep
    FOR VALUES FROM ('2026-06-01') TO ('2026-07-01');

When to partition: table over 100 GB or 100M rows, or growing toward that. Below that the complexity is not worth it.
Key > Method: the partition key determines which queries can prune and which unique constraints are possible. RANGE on created_at is the common default for append-only tables.
Packages: django-postgres-extra for native Django-style partition ops, pg_partman for scheduled maintenance, psycopack for online table conversion.
