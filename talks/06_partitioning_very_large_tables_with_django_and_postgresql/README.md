# Partitioning Very Large Tables with Django and PostgreSQL

> **Speaker:** Tim Bell — Kraken Tech. DBA by accident; the database-heavy talks lately (locking, bigint migrations, now this) all come from work on a 9 billion row table.
> **Event:** DjangoCon Europe 2026

## What is this about?

Postgres normally hides how it stores data. Tables grow, indexes work, autovacuum runs — you just use SQL. But when a single table reaches billions of rows and terabytes, the abstraction leaks: vacuums take 20 hours, block schema changes, and a weekly vacuum cycle starts to limit what you can do. **Partitioning** splits one logical table into many physical tables on a `partition key`, so each partition is vacuumed, indexed and queried independently. This talk is about how to pick the partition key, how Postgres executes queries against partitioned tables, and the bits of Django 5.2 you need to make it work.

## The Problem

- Kraken had a workflow-steps table: **9 billion rows, 3 TB, 20-hour vacuums** running about once a day
- Long vacuums **blocked the schema changes** they needed for a bigint migration
- "Delete old data" / "start a new table" both push complexity into the app — not generally applicable
- Autovacuum is per-table, so the only way to vacuum in parallel is to have more tables

## The Solution

Native Postgres partitioning: one **partitioned parent** that is a virtual table with no storage, and N **child partitions** that are ordinary tables with their own indexes, stats, and vacuum lifecycle. Rows are routed to a child based on the value of the **partition key**.

Three partitioning methods — pick one:

| Method | When to use | Example |
|---|---|---|
| **RANGE** | Time-series, monotonically growing data | `PARTITION BY RANGE (created_at)` |
| **LIST** | Fixed small set of values | `PARTITION BY LIST (region)` |
| **HASH** | No natural grouping, just want parallelism | `PARTITION BY HASH (customer_id)` |

### The two things that actually matter

1. **The partition key must be part of every unique constraint** — including the primary key. Postgres implements uniqueness via per-partition indexes, and there's no cheap way to enforce uniqueness across partitions. So partitioning on `created_at` forces a composite `(id, created_at)` PK.
2. **Partition pruning** only kicks in when the query's `WHERE` includes the partition key. Without that filter, Postgres scans every partition. The partition key should be present in 90%+ of your hot queries.

### How Django fits in

- Django's schema editor can't emit `PARTITION BY RANGE (...)` — create the table with `RunSQL` in a migration, or use a helper package. The model should be `managed = False` so Django doesn't fight you.
- **Django 5.2's `CompositePrimaryKey`** unlocks the composite-PK requirement — reads through the ORM work transparently.
- The leaky-abstraction moment: Django 5.2 composite PKs don't combine with `BigAutoField`, so your `id` comes from a database-side `IDENTITY` sequence and ORM writes need a tiny bit of raw SQL.

### Existing packages

- [`django-postgres-extra`](https://github.com/SectorLabs/django-postgres-extra) — native Django-style migration operations for `PartitionedPostgresModel`. Nice ergonomics, but no "convert existing table" path.
- [`pg_partman`](https://github.com/pgpartman/pg_partman) — Postgres extension, language-agnostic, popular for maintenance (creating/dropping partitions on a schedule). Not Django-aware.
- [`psycopack`](https://github.com/octoenergy/psycopack) — Kraken's in-house tool for replicating a table + making schema changes online. Being extended to do online conversion of existing tables into partitioned ones.

## How to Use It

### Install

```bash
pip install "Django>=5.2" "psycopg[binary]"
```

### 1. Create the partitioned table (raw SQL in a migration)

```python
# workflows/migrations/0001_initial.py
from django.db import migrations

class Migration(migrations.Migration):
    initial = True
    operations = [
        migrations.RunSQL(
            sql="""
            CREATE TABLE workflows_workflowstep (
                id          BIGINT      GENERATED BY DEFAULT AS IDENTITY,
                workflow_id BIGINT      NOT NULL,
                name        VARCHAR(100) NOT NULL,
                status      VARCHAR(20) NOT NULL,
                created_at  TIMESTAMPTZ NOT NULL,
                -- partition key must be part of every unique constraint
                PRIMARY KEY (id, created_at)
            ) PARTITION BY RANGE (created_at);

            CREATE TABLE workflows_workflowstep_2026_04
                PARTITION OF workflows_workflowstep
                FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');

            CREATE TABLE workflows_workflowstep_default
                PARTITION OF workflows_workflowstep DEFAULT;
            """,
            reverse_sql="DROP TABLE IF EXISTS workflows_workflowstep CASCADE;",
        ),
    ]
```

### 2. Declare the model with a composite PK

```python
# workflows/models.py
from django.db import models

class WorkflowStep(models.Model):
    pk = models.CompositePrimaryKey("id", "created_at")
    id = models.BigIntegerField()
    created_at = models.DateTimeField()
    workflow_id = models.BigIntegerField()
    name = models.CharField(max_length=100)
    status = models.CharField(max_length=20)

    class Meta:
        managed = False     # the SQL migration owns the schema
        db_table = "workflows_workflowstep"
```

### 3. Read through the ORM — it just works

```python
# Partition pruning kicks in because created_at is in the WHERE.
WorkflowStep.objects.filter(
    created_at__gte=date(2026, 4, 1),
    created_at__lt=date(2026, 5, 1),
    workflow_id=7,
)
# EXPLAIN: scans only workflows_workflowstep_2026_04
```

### 4. Write via raw SQL (the leaky bit)

```python
with connection.cursor() as cur:
    cur.execute(
        """
        INSERT INTO workflows_workflowstep
            (workflow_id, name, status, created_at)
        VALUES (%s, %s, %s, %s);
        """,
        [7, "validate_meter", "pending", created_at],
    )
```

## Experiment

This folder is a runnable Django project with a partitioned workflow-steps table, 7 monthly partitions, a DEFAULT partition, and management commands that demonstrate the mechanics end-to-end. **Requires PostgreSQL.**

```bash
# Postgres in docker (exposes 55432 on the host — no collision with local PG)
docker compose up -d
docker compose exec db pg_isready -U partitioning   # wait for "accepting connections"

pip install -r requirements.txt
python manage.py migrate

python manage.py seed_data           # ~3500 rows across 7 monthly partitions + 1 in DEFAULT
python manage.py show_partitions     # row counts and sizes per partition
python manage.py demo_pruning        # EXPLAIN with partition-key filter → 1 partition scanned
python manage.py demo_pruning --no-partition-key   # EXPLAIN without → Append over all partitions
python manage.py demo_orm            # regular Django querysets against the partitioned table
python manage.py add_partition 2026-06
python manage.py runserver
```

Routes:

| `/test/` | Interactive test panel — partition viewer, pruning demo, ORM queries |

What each management command shows:

| Command | Demonstrates |

## Key Takeaways

- **Partition when a table is >100 GB or >100 M rows**, or is growing toward that. Below that the complexity isn't worth it.
- **Partition key > partitioning method.** The key determines which queries can prune and which unique constraints are possible. RANGE on `created_at` is the common default for append-only tables.
- **Every unique constraint — including the PK — must contain the partition key.** That's why you end up on composite primary keys.
- **Partitioning is transparent to Django reads.** `filter()`, aggregates, ordering all work. The ORM does not need to know about partitions.
- **Django can't emit `PARTITION BY` yet.** Use `RunSQL` in migrations (this demo), or `django-postgres-extra` for native-looking migration operations.
- **Converting an existing large table online** is its own project. Don't attempt it with raw SQL; look at `pg_partman` for scheduled maintenance and `psycopack` for the conversion itself.

## Q&A Highlights

- **What about foreign keys into a partitioned table?** Allowed — FKs referencing the partitioned parent work. FKs *from* a partitioned table work too, with the usual partition-key caveats on uniqueness of the target.
- **Can you create new partitions with Django migrations?** Yes — a `RunSQL` migration per month is fine. For many installations you'll want a scheduled job (pg_partman, or a cron'd management command) instead of a migration-per-partition.
- **Is the DEFAULT partition safe in production?** It works but makes operations painful: attaching a new partition whose range overlaps rows already in DEFAULT requires detaching DEFAULT first, moving matching rows, then reattaching. Most teams avoid DEFAULT once they have partition-creation automation in place.
- **How does vacuum behave?** Each partition is vacuumed independently by autovacuum, with its own thresholds. That's the whole point — a 3 TB vacuum becomes thirty 100 GB vacuums that can run in parallel.
